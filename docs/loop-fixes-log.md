# Лог автономных правок (/loop)

Журнал мелких исправлений логики и багов, найденных в автономном режиме `/loop`.
Каждая итерация — одна правка с обоснованием. Один общий файл на все операции.

---

## Итерация 1 — 2026-06-10

**Файл:** `src/components/ProductCard.tsx`
**Тип:** DRY / дублирование логики

**Проблема:** В `ProductCard` минимальная цена вычислялась вручную
(`Object.values(perfume.prices).filter(...).then Math.min`) — точная копия тела
`getMinPrice()`, которую ввели в этапе 1 миграции именно ради устранения таких
дублей. Остальные потребители (`CatalogClient`, `product/[id]`) уже использовали
`getMinPrice`, а `ProductCard` остался с копипастой.

**Исправление:** заменил ручной расчёт на `getMinPrice(perfume)` из `@/data/utils`.
Поведение идентично (та же фильтрация `> 0` и null при отсутствии цен), но логика
теперь в одном месте — при изменении правила «минимальной цены» не разъедется.

**Проверка:** `npm run build` — зелёный, 16/16 страниц, типы валидны.

## Итерация 2 — 2026-06-10

**Файл:** `src/contexts/CartContext.tsx`
**Тип:** Логическая ошибка / устойчивость к нечисловому вводу

**Проблема:** `updateQuantity(id, vol, quantity)` не проверял `quantity` на
конечность. При `NaN` (например, из пустого `<input type="number">` или битого
парсинга): `NaN < 1` → `false` (позиция не удаляется), `Math.min(Math.floor(NaN),
MAX)` → `NaN`. Позиция получала `quantity: NaN`, после чего `total`/`count`
становились `NaN` → вся корзина показывала «NaN ₽». При этом соседние методы того
же файла (`addItem`, `sanitizeStoredItems`) уже защищены `Number.isFinite` —
налицо несогласованность. Сегодня вызовы идут только через кнопки ±1 (NaN не
приходит), но метод публичный и сигнатура допускает любой `number`.

**Исправление:** добавил `!Number.isFinite(quantity)` в условие очистки позиции,
рядом с `quantity < 1`. Нечисловой ввод теперь безопасно удаляет позицию вместо
заражения корзины NaN. Согласовано с защитой в остальном файле.

**Проверка:** `npm run build` — зелёный, типы валидны.

## Итерация 3 — 2026-06-10

**Файл:** `src/components/CatalogClient.tsx`
**Тип:** Логическая ошибка / тихо неработающий фильтр

**Проблема:** Границы фильтра цены парсились как `priceMin ? parseInt(priceMin) : 0`
(без radix, без проверки NaN). При частичном/битом вводе в `<input type=number>`
(`"1e"`, вставка, и т.п.) `parseInt` мог вернуть `NaN`. Тогда в условиях фильтра
`minVal > 0` → `NaN > 0` → `false` и `maxVal < Infinity` → `false`: фильтр по цене
**молча не применялся**, хотя чип «фильтр активен» показывался (`priceActive` =
true при непустой строке). Пользователь думает, что фильтрует по цене, а
показываются все товары — рассинхрон состояния и поведения.

**Исправление:** безопасный парс — `parseInt(x, 10)` + `Number.isFinite(...) && > 0`,
иначе граница считается «не заданной» (0 / Infinity). Битый ввод больше не ломает
фильтр молча. Radix 10 добавлен явно. Согласовано с NaN-защитой из итерации 2.

**Проверка:** `npm run build` — зелёный, 19/19 страниц, типы валидны.

## Итерация 4 — 2026-06-10

**Файл:** `src/data/utils.ts`
**Тип:** Логическая ошибка / регрессия, открывшаяся после этапа 2 (админ-бот)

**Проблема:** `slugifyBrand` не транслитерировал кириллицу:
`brand.replace(/[^a-z0-9]+/g, '-')` выкидывал все кириллические символы. Для
бренда кириллицей это давало пустой или усечённый slug:
- `"Сибирь"` → `""` → `buildBrandEntries` отфильтровывает (slug.length>0) →
  бренд **исчезает** из списка/фильтра, страница бренда → `notFound()`.
- `"Brocard Москва"` → `"brocard"` → кириллическая часть молча теряется (риск
  коллизий разных брендов в один slug).

Стало критично после этапа 2: админ-бот теперь умеет добавлять бренды, и
кириллический бренд получил бы нерабочую страницу/фильтр. Вдобавок бот генерирует
`id` аромата через свою `slugify` *с* транслитом (`workers/admin-bot.js`), а сайт
считал бренд-slug *без* — рассинхрон логики.

**Исправление:** добавил транслитерацию кириллицы той же таблицей, что в
`admin-bot.js`. Латинские бренды не изменились (`Tom Ford`→`tom-ford` —
обратная совместимость), кириллические дают рабочий slug (`Сибирь`→`sibir`,
`Новая Заря`→`novaya-zarya`).

**Проверка:** `npm run build` — зелёный, 19/19. Прогон по примерам подтвердил
сохранность латинских slug и корректность кириллических.
