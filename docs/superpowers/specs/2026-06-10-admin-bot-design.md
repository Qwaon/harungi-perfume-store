# Дизайн: Telegram админ-бот каталога (этап 2 миграции на Supabase)

**Дата:** 2026-06-10
**Статус:** дизайн утверждён, готов к написанию плана
**Зависит от:** этап 1 (каталог в Supabase, таблица `perfumes`) — завершён.
**Роадмап:** `docs/superpowers/specs/2026-06-07-supabase-migration-roadmap.md` (этап 2).

## Цель

Отдельный Telegram-бот для администратора: добавлять, редактировать, удалять
ароматы и загружать фото — запись напрямую в Supabase. Заменяет ручное
редактирование таблицы. Полный CRUD + загрузка изображений в Supabase Storage.

## Утверждённые решения (брейншторм 2026-06-10)

| Развилка | Решение |
|---|---|
| Объём | Полный CRUD (add/edit/delete) + загрузка фото в Storage |
| Хостинг | Отдельный Cloudflare Worker `workers/admin-bot.js` (рядом с `order-webhook.js`) |
| Бот | **Отдельный** бот-токен (не тот, что шлёт заказы) |
| Авторизация | Allowlist Telegram user id (env `ADMIN_USER_IDS`) |
| UX ввода | Пошаговый диалог (поле за полем) |
| Состояние диалога | Таблица Supabase `admin_sessions` |
| Редактирование | По одному полю через меню (не весь диалог заново) |
| id аромата | Автослаг из brand+name, транслитерация кириллицы, суффикс при коллизии |

## Архитектура

Новый Cloudflare Worker, изолированный от приёма заказов. Один файл
`workers/admin-bot.js`, логически разделён на слои:

1. **Роутинг + авторизация** — принимает Telegram webhook (POST), проверяет
   webhook-секрет и `from.id` против allowlist. Чужих молча игнорирует.
2. **FSM (машина диалога)** — читает сессию из `admin_sessions`, по шагу решает
   что спросить/сохранить, пишет сессию обратно. Ядро — чистая функция перехода.
3. **Supabase-слой** — `selectPerfumes`, `insertPerfume`, `updatePerfume`,
   `deletePerfume`, `uploadImage`, `deleteImages` (REST + Storage API).
4. **Telegram-слой** — обёртка `tg(method, body)`, клавиатуры, рендер карточки
   аромата (как будет на сайте).

**Изоляция:**
- Отдельный Worker, отдельный деплой, отдельный бот-токен. `order-webhook.js`
  не затрагивается — если админка упадёт, приём заказов продолжает работать.
- У админ-бота — **`service_role`** ключ Supabase (полный доступ, обходит RLS),
  только в env этого Worker'а. Публичный сайт (этап 1) продолжает читать через
  `publishable` ключ + read-only RLS.

**Связь с публичным каталогом:** бот пишет в ту же таблицу `perfumes`. Сайт
подхватывает изменения через ISR `revalidate=60` (задержка до минуты), отдельных
действий не требуется.

## Схема данных (новое в Supabase)

К существующей таблице `perfumes` (этап 1) добавляются:

### Таблица `admin_sessions`

```sql
create table admin_sessions (
  tg_user_id  bigint primary key,   -- один активный диалог на админа
  flow        text,                 -- 'add' | 'edit' | 'delete'
  step        text,                 -- текущий шаг: 'name','brand','price_5ml',...
  draft       jsonb default '{}',   -- накопленные поля
  target_id   text,                 -- id аромата при edit/delete (null при add)
  last_update_id bigint,            -- idempotency: последний обработанный update_id
  updated_at  timestamptz default now()
);
alter table admin_sessions enable row level security;  -- БЕЗ политик: только service_role
```

RLS включён без единой политики → `publishable`-ключ сайта эту таблицу не видит.
Сессия старше 24ч считается протухшей.

### Storage-бакет `perfume-images`

- Публичный на чтение (фото открыты по прямой ссылке, как сейчас Unsplash).
- Бот заливает фото, получает URL
  `https://<ref>.supabase.co/storage/v1/object/public/perfume-images/<id>-<n>.jpg`.
- URL дописывается в колонку `images` (CSV) таблицы `perfumes` — формат, который
  сайт уже читает (этап 1, `parseList`). **Фронт не меняется.**

### Чего НЕ делаем

Нормализацию (отдельные таблицы цен/нот). Остаёмся на плоской схеме этапа 1 —
бот пишет в те же колонки `price_5ml`, `notes_top` (CSV) и т.д. Совместимость с
`transformRow` без изменений.

## UX диалога

Справочные значения (из `src/types/index.ts`):
- `gender`: мужской / женский / унисекс
- `scentType`: цветочный / восточный / древесный / свежий / фужерный / шипровый / гурманский
- `format`: оригинал / распив
- цены: `price_5ml`, `price_10ml`, `price_15ml`, `price_20ml`, `price_original`

**Главное меню** (`/start` или команда): inline-кнопки
`➕ Добавить` · `📋 Список / Редактировать`. По тапу на товар — `✏️ Изменить · 🗑 Удалить`.

### Поток «Добавить» (пошагово)

1. `name` — текст (обязательно)
2. `brand` — текст (обязательно)
3. `description` — текст / «Пропустить»
4. `gender` — кнопки
5. `scentType` — кнопки
6. `format` — кнопки
7. `price_5ml … price_20ml`, `price_original` — число / «Пропустить» (пропуск =
   объёма нет). Нечисло → переспрос того же шага.
8. `notes_top / middle / base` — текст через запятую / «Пропустить»
9. `original_volume_ml` — **только если** была цена `original`
10. **Фото** — админ шлёт изображения (можно несколько подряд), затем «Готово».
    Каждое заливается в Storage. Или «Пропустить».
11. Флаги `inStock / featured / newArrival / bestseller` — кнопки-тумблеры.
12. **Превью + подтверждение** — рендер карточки + `✅ Сохранить · ✖️ Отмена`.
    Запись в `perfumes` только по «Сохранить».

**id** генерируется автоматически: slugify(brand + name), транслитерация
кириллицы в латиницу. При коллизии с существующим id — суффикс `-2`, `-3`, …

### Поток «Редактировать»

Выбор товара → бот показывает **список полей кнопками** → тап по полю → бот
спрашивает новое значение → сразу `update`. Весь диалог заново не гоняется.

### Поток «Удалить»

Выбор товара → превью → `⚠️ Точно удалить? Да / Нет`. По «Да» — `delete` +
удаление связанных фото из Storage.

**Везде:** `/cancel` сбрасывает сессию. «← Назад» на шагах добавления —
на предыдущий шаг.

## Безопасность

- **Webhook-секрет** — `X-Telegram-Bot-Api-Secret-Token` = env
  `ADMIN_WEBHOOK_SECRET`. Чужой POST → 401.
- **Allowlist** — `from.id` ∈ `ADMIN_USER_IDS` (CSV в env). Не в списке → бот
  молчит (не отвечает, чтобы не выдавать существование).
- **`service_role` ключ** — только в env Worker'а. Не в гите, не во фронте, не
  логируется.

## Обработка ошибок и краевые случаи

- Сбой Supabase (запись/Storage) → понятная ошибка админу + **draft сохранён** в
  сессии, можно повторить «Сохранить».
- Дубль апдейта от Telegram (ретрай) → idempotency по `last_update_id` в сессии.
- Невалидный ввод → переспрос того же шага, draft цел.
- Брошенный диалог → сессия живёт; `/start` предлагает продолжить или начать
  заново. Сессия >24ч — протухла.
- Не-изображение в шаге фото → отклоняем, просим картинку.
- Удаление аромата из истории заказов (этап 3) не трогается — удаляем только из
  каталога.
- Частичная загрузка фото (3 из 5) → сохраняем залитое, сообщаем сколько.

## Тестирование

**Чистые функции — `node --test`** (встроенный раннер, без новых зависимостей),
отдельный файл рядом с Worker'ом:
- `slugify(brand, name)` — корректный id, транслитерация кириллицы, коллизии.
- FSM-переход `(session, input) → { nextStep, draftPatch, reply }` — ядро диалога.
- `validateField(step, value)` — ok/ошибка (цена число, обязательные не пустые).
- `draftToPerfumeRow(draft)` — объект для вставки (имена колонок, CSV-сборка).

**Интеграция — ручной чеклист** после деплоя:
1. Не-админ пишет → молчание.
2. Админ `/start` → меню.
3. Добавить аромат с фото → появился на сайте после revalidate.
4. Отредактировать цену → изменилась на сайте.
5. Скрыть (`inStock=false`) → пропал из выдачи.
6. Удалить → пропал, фото убрано из Storage.
7. Невалидный ввод, `/cancel`, брошенный диалог.

Сетевой Supabase/Telegram-слой тестами не покрываем — проверяется чеклистом.

## Деплой и привязка (разово, вне кода — делает владелец по инструкции)

1. Создать бота в @BotFather, получить токен.
2. Env в Cloudflare (новый Worker): `ADMIN_BOT_TOKEN`, `ADMIN_USER_IDS`,
   `ADMIN_WEBHOOK_SECRET`, `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`.
3. Создать таблицу `admin_sessions` и бакет `perfume-images` в Supabase (SQL +
   Storage UI; SQL войдёт в репозиторий как `supabase/002_admin.sql`).
4. `setWebhook` на URL нового Worker'а с `secret_token`.
