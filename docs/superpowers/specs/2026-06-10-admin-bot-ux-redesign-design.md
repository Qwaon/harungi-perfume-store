# Дизайн: Юзабельный редизайн навигации админ-бота каталога

**Дата:** 2026-06-10
**Статус:** дизайн утверждён, готов к написанию плана
**Зависит от:** `workers/admin-bot.js` (этап 2 миграции, уже на проде).
**Базовый дизайн:** `docs/superpowers/specs/2026-06-10-admin-bot-design.md`.

## Цель

Сделать админ-бота каталога юзабельным. Текущая реализация работает, но
неудобна: каждое действие шлёт новое сообщение (сообщения накапливаются,
список «застревает» вверху), нет кнопки «Назад», нет постоянных команд,
фото аромата не показывается реальным изображением, правка ограничена 7 полями.

## Проблемы (подтверждены по коду `workers/admin-bot.js`)

| # | Проблема | Где в коде |
|---|---|---|
| 1 | Каждое действие = новое `sendMessage`, ничего не редактируется/удаляется → сообщения копятся, список остаётся вверху | весь диспетчер, `sendMessage` (L295) |
| 2 | Нет кнопки «Назад» нигде (list → pick → card) | L369–379 |
| 3 | Фото не отображается — карточка только текст (`renderCard`), изображение не шлётся | L586, `renderCard` |
| 4 | Нет постоянных команд — только `/start` и `/cancel`, нет меню команд | L333–337 |
| 5 | Список выводит все ароматы кнопками сразу — не масштабируется на 50+ | L362–367 |
| 6 | Правка только 7 полей; нельзя править остальные цены, enum-поля, ноты, фото | L401 |
| 7 | Кнопки «Нет»/«Отмена» уходят в немой `noop` — оставляют устаревшие сообщения | L386, L505 |

## Утверждённые решения (брейншторм 2026-06-10)

| Развилка | Решение |
|---|---|
| Модель навигации | **Одно сообщение, редактируется на месте** (list → card → edit → back перерисовывают то же сообщение). Стандартный паттерн админ-панели Telegram |
| Постоянные команды | `/start`, `/add`, `/list`, `/cancel` — как текст-команды в коде + меню BotFather (`setMyCommands`) |
| Список при росте каталога | **Пагинация по 8** на страницу (`‹ Назад` / `Далее ›` + счётчик) |
| Правка существующего | **Все поля** (цены, ноты, пол/тип/формат) + **управление фото** + **все 4 флага** |
| «Назад» внутри потока add | **Нет** — достаточно `/cancel` (поток add линейный, его не трогаем) |

## Архитектура

Переработка `workers/admin-bot.js`. Вводим понятие **«экран»**: каждый экран —
это `{ text, keyboard }`, который рисуется **редактированием** текущего
меню-сообщения (Telegram `editMessageText`), а не отправкой нового.

### 1. Слой экранов

- **Чистая функция `buildScreen(screen, data)` → `{ text, reply_markup }`** —
  строит текст и клавиатуру для экрана по его типу и данным. Тестируемая,
  без сети. Типы экранов:
  - `menu` — главное меню (➕ Добавить / 📋 Список)
  - `list` — страница списка (до 8 кнопок ароматов + `‹`/`›` пагинация +
    `🏠 В меню`)
  - `card` — карточка аромата (текстовый вариант, если фото нет)
  - `editmenu` — меню полей правки (все колонки, сгруппированы) +
    `‹ К карточке`
- **Хелпер `editScreen(userId, menuMessageId, screen, env)`** — вызывает
  `editMessageText` с результатом `buildScreen`. Если редактирование не
  удалось (сообщение слишком старое / удалено) — фолбэк на `sendMessage` и
  обновление `menu_message_id`.
- **`menu_message_id`** текущего меню-сообщения хранится в сессии — новая
  колонка `menu_message_id bigint` в `admin_sessions` (миграция
  `supabase/002_admin.sql`).

### 2. Фото-карточка (обход ограничения Telegram)

Telegram **не позволяет** редактировать текстовое сообщение в фото-сообщение и
обратно. Поэтому карточка с фото — особый случай:

- Открытие `card:<id>`, **если у аромата есть фото:**
  бот **удаляет** меню-сообщение и шлёт `sendPhoto` (первое фото) с
  подписью-карточкой (`renderCard`) и кнопками
  (`✏️ Изменить`, `🗑 Удалить`, `‹ К списку`, `🏠 В меню`).
  `message_id` фото-сообщения сохраняется в `menu_message_id`.
- Уход с карточки (Назад / в меню / правка): фото-сообщение **удаляется**,
  следующий экран рисуется новым сообщением, его id снова в сессию.
- Если фото **нет** — карточка обычным текстовым экраном (правится на месте).
- Новые хелперы: `sendPhotoScreen()`, `deleteMessage()`.

Это также **диагностирует** жалобу «фото не отображается»: если после
показа реальным `sendPhoto` фото всё равно не видно, значит URL в колонке
`images` битый (бакет не публичный / неверный путь) — это станет видно сразу.
В рамках работы проверяем формат URL из `uploadImage` против публичного
бакета `perfume-images`.

### 3. Расширенная правка (`editmenu`)

Меню правки покрывает **все** колонки (не 7):

- Текстовые/ценовые поля (`name`, `brand`, `description`, все `price_*`,
  `original_volume_ml`, `notes_*`) → ввод значения (`editval:<field>`,
  как сейчас).
- enum-поля (`gender`, `scentType`, `format`) → кнопки-варианты
  (переиспользуем `choiceKeyboard` + `validateField`).
- Все 4 флага (`inStock`, `featured`, `newArrival`, `bestseller`) →
  тумблер сразу (как сейчас inStock/featured, расширяем на 4).
- **Фото:** `➕ Добавить фото` (принять изображения, дописать в `images`
  существующего аромата через тот же `downloadAndStorePhoto` + PATCH) и
  `🗑 Очистить фото` (обнулить `images` + `deleteImages`).
- После любой правки — **возврат на карточку** (контекст не теряется),
  сессия не сбрасывается.

### 4. Постоянные команды

- В коде (работают всегда, даже посреди потока):
  `/start` → menu, `/add` → старт потока add, `/list` → list (страница 0),
  `/cancel` → сброс сессии + menu.
- Разово (документируется curl в CLAUDE.md): `setMyCommands` с этими четырьмя —
  появляются в меню «/» рядом с полем ввода.

### 5. Чистка `noop`

Кнопки «Нет» / «Отмена» больше не уходят в немой `noop` — ведут на осмысленный
экран (назад к карточке / в меню), перерисовывая текущее сообщение.

## Что НЕ трогаем (YAGNI)

- **Поток `add`** — линейный пошаговый диалог остаётся как есть. Шаг «Назад»
  внутри add не добавляем (усложнил бы FSM с условными пропусками); `/cancel`
  достаточно.
- **Поиск по списку** — отклонён, пагинации по 8 хватает.
- Сигнатуры существующих чистых функций (`slugify`, `validateField`,
  `advanceAdd`, `draftToPerfumeRow`, FSM `add`) — **без изменений**.

## Тестирование

- **Новые чистые функции** покрываются `node --test workers/admin-bot.test.mjs`:
  - `paginate(items, page, size)` → срез страницы + есть ли prev/next.
  - `buildScreen(screen, data)` → корректные текст и клавиатура для
    `menu` / `list` / `card` / `editmenu`.
  - `editFieldsKeyboard()` / группировка полей правки.
- **Существующие тесты** не ломаются (сигнатуры не тронуты).
- **Ручная проверка** в Telegram по чек-листу: навигация туда-обратно,
  показ фото, удаление фото-сообщения при уходе, правка каждого типа поля,
  пагинация на >8 ароматах, команды `/add` `/list` посреди потока.

> ⚠️ Сетевые/Telegram-вызовы (`editMessageText`, `sendPhoto`, `deleteMessage`,
> Supabase REST) не юнит-тестируются — проверяются вручную в боте.

- `workers/admin-bot.js` — переработка диспетчера и слоя рендера.
- `workers/admin-bot.test.mjs` — новые тесты чистых функций.
- `supabase/002_admin.sql` — колонка `menu_message_id bigint`.
- `CLAUDE.md` — раздел админ-бота: навигация одним сообщением, `setMyCommands`,
  команды `/add` `/list`.
- (этот спек) — дизайн редизайна.

## Открытые риски

- **Telegram rate-limit** на частые `editMessageText` — навигация кликами
  редкая, в лимиты не упирается.
- **Старое меню-сообщение** (>48ч) нельзя редактировать → фолбэк на
  `sendMessage` в `editScreen`.
- **Гонка** при двойном клике — снимается существующей idempotency по
  `last_update_id`.
