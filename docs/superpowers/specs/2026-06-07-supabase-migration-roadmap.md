# Роадмап: миграция на Supabase (каталог + админ-бот + заказы)

**Дата:** 2026-06-07
**Статус:** утверждённое направление, реализация отложена

## Цель и мотивация

Перевести HARUNGI с Airtable на Supabase (Postgres) как единый источник правды, добавить Telegram-бот администрирования каталога и перенести туда же Order CRM.

Мотивы (со слов владельца):
- **Управление через бот** — добавлять/редактировать ароматы из Telegram, а не руками в таблице.
- **Единая база** — каталог + заказы + история в одном месте, без зоопарка Airtable / perfumes.json / Telegram CloudStorage.
- **Задел на рост** — настоящая БД под будущие фичи (аккаунты, аналитика, склад).

### Почему Supabase лучше Airtable для этих целей

- Один Postgres под каталог и заказы с настоящими связями и транзакциями (то, что в Order CRM пришлось упростить «составом-текстом», в Postgres делается нормально).
- Админ-бот = CRUD; через Postgres/REST это проще и надёжнее, чем Airtable API (лимит 5 req/s, нет транзакций).
- Row Level Security (роль админ vs публичный read-only) и realtime из коробки.
- Бесплатный тариф щедрее; нет лимита 1000 записей/база Airtable.

### Компромисс, который надо помнить

После переезда каталога Airtable перестаёт быть источником правды → **удобное редактирование пропадает до готовности админ-бота**. Промежуток закрывается через Supabase Table Editor (этап 1) и затем полностью — ботом (этап 2).

## Решение по ведению: 3 под-проекта подряд

Делаем как три самостоятельных этапа (каждый — рабочий результат), один за другим без долгих пауз. Не один монолит — чтобы не остаться без редактируемого каталога в середине и чтобы баги локализовались по этапам. Каждый этап проходит свой цикл spec → план → реализация.

## Этап 1 — Каталог Airtable → Supabase (только чтение)

**Статус:** дизайн утверждён (Вариант A), реализация отложена.

**Цель:** сайт читает каталог из Supabase вместо Airtable. Страницы, компоненты и тип `Perfume` не меняются.

### Утверждённые решения

- **Чтение:** Supabase REST API с `next: { revalidate: 60 }` (ISR сохраняется). Без `@supabase/supabase-js`.
- **Схема Postgres:** один-в-один с Airtable. Таблица `perfumes` с теми же плоскими колонками (`price_5ml`, `price_10ml`, `price_15ml`, `price_20ml`, `price_original`, `notes_top/middle/base` как CSV-текст, `gender`, `scentType`, `format`, `images` CSV, чекбоксы `inStock/featured/newArrival/bestseller`, `original_volume_ml`). Нормализацию (отдельные таблицы цен/нот) НЕ делаем сейчас.
- **Фолбэк:** убрать фолбэк на `perfumes.json`. Полагаемся на Supabase (надёжнее Airtable). ⚠️ Риск: при сбое каталог пустой — в плане предусмотреть мягкую деградацию (ошибка/последний ISR-кеш вместо пустой страницы), обсудить при написании плана.
- **Миграция данных:** разовая заливка из `perfumes.json` в Supabase. Перед заливкой один раз освежить JSON (`npm run sync-catalog`).

### Архитектура (Вариант A — заменить источник, сохранить интерфейс)

- Меняется **только** `src/data/catalog.ts`: `fetchAirtableRecords()` → `fetchSupabaseRows()` (GET Supabase REST, ISR). Публичный интерфейс (`getPerfumes`, `getBrands`, `getBrandEntries`, `getPerfumesByBrandSlug`) и тип `Perfume` без изменений.
- `src/data/transform.ts` адаптируется: вместо `record.fields[...]` читает поля строки напрямую (`row[...]`). Логика парсинга цен/нот/листов — та же.
- `src/data/utils.ts` (`enrichPerfume`, `buildBrandEntries`) — без изменений.
- `enrichPerfume` (вывод occasion/season/intensity/sourceType) и фильтр `inStock` сохраняются.

### Env (этап 1)

- `SUPABASE_URL`, `SUPABASE_KEY` (server-only, без `NEXT_PUBLIC_`). В Vercel Production — те же.
- Airtable-переменные пока остаются (для сравнения/отката), удаляются после стабилизации.

### Затрагиваемые файлы

- `src/data/catalog.ts` — источник чтения.
- `src/data/transform.ts` — маппинг строки Supabase.
- Скрипт миграции `perfumes.json` → Supabase (разовый, в `scripts/`).
- (Опционально позже) обновить/заменить `scripts/sync-catalog.mjs`.

### Вне рамок этапа 1

Запись в Supabase, админ-бот, перенос Order CRM, нормализация схемы.

## Этап 2 — Админ-бот (управление каталогом из Telegram)

**Статус:** не спроектирован (брейншторм отдельно перед реализацией).

**Цель:** отдельный Telegram-бот для администратора: добавлять, редактировать, скрывать (`inStock`), помечать `featured/newArrival/bestseller` ароматы — запись в Supabase.

### Открытые вопросы для будущего брейншторма

- **Хостинг бота:** Cloudflare Worker (как order-webhook) vs отдельный сервис. Long polling vs webhook.
- **Авторизация админа:** allowlist по Telegram user id; роль/RLS в Supabase.
- **UX ввода аромата:** пошаговый диалог (поле за полем) vs форма/Mini App vs структурированное сообщение. Загрузка изображений (URL vs upload в Supabase Storage).
- **Запись в Supabase:** service role key только на стороне бота (server-only). Валидация на уровне БД (constraints) + в боте.
- **Выделить переиспользуемый Supabase-слой** (идея из брейншторма этапа 1, Вариант B): низкоуровневый клиент/запросы для бота и заказов — оформить здесь, увидев реальные потребности записи.
- **Связь с публичным каталогом:** запись бота → ISR revalidate=60 подхватит автоматически; при необходимости — on-demand revalidation.

### Зависит от

Этап 1 (рабочая таблица `perfumes` в Supabase + схема).

**Цель:** заказы и история — в Supabase вместо Airtable `Orders` и Telegram CloudStorage.

- **История заказов в Mini App:** сейчас в CloudStorage (`lib/orderHistory.ts`). Решить — оставить локально или читать из Supabase по `tgUserId`.
- **Аналитика/статусы:** Supabase даёт нормальные запросы/вьюхи для отчётов по продажам.
