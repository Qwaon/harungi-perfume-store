# Admin Panel Improvements — Design

## Контекст

Текущая веб-админка (`/admin`) минимальна: дашборд — голые счётчики заказов и
товаров, каталог — список карточек с поиском по названию/бренду без фильтров
и сортировки, изменение цены — только через форму редактирования одного товара.

Главная боль бизнеса: цены на 5/10/15/20мл и оригинал у части или у всех
товаров периодически меняются (поставщики поднимают цены), и редактировать
их по одному долго. Также нужны более гибкие сортировка/группировка/фильтры
по каталогу и более информативный дашборд.

## 1. Каталог — фильтры, сортировка, группировка, чекбоксы

`CatalogListClient` (src/components/admin/CatalogListClient.tsx) остаётся
списком карточек-строк (фото + бренд/название + статус), с добавлениями:

- **Чекбокс** слева на каждой строке + панель массового выбора сверху:
  «Выбрать все» / «Снять всё», счётчик выбранных.
- **Цены в строке**: компактный вывод 5/10/15/20/ориг (только заполненные,
  например `5мл 1200 · 10мл 2200`).
- **Фильтры** (выпадающие/чипы над списком): наличие (`inStock`),
  `gender`, `scentType`, `format`, флаги `featured`/`newArrival`/`bestseller`.
  Фильтры комбинируются (AND), применяются к уже загруженному списку (client-side).
- **Сортировка** (select): по бренду (А-Я), по названию (А-Я), по наличию
  (сначала в наличии), по цене (по `price_5ml`, если пусто — в конец).
- **Группировка по бренду**: при сортировке «по бренду» рендерятся
  sticky-заголовки с именем бренда над каждой группой строк.
- **URL-параметр `alert`**: при монтировании читается `?alert=no-photo|no-price|out-of-stock`
  из query string и применяется как предустановленный фильтр (переход с дашборда).

Поиск по названию/бренду остаётся, применяется поверх фильтров.

## 2. Массовое изменение цен

### UI: `BulkPriceModal` (новый компонент `src/components/admin/BulkPriceModal.tsx`)

Открывается кнопкой «Изменить цены (N)», активной при ≥1 выбранном товаре
(или при выборе «выбрать все», что выбирает все товары, прошедшие текущие
фильтры — действие происходит над **видимым после фильтрации** списком).

Форма:
- Чипы выбора полей: `5мл`, `10мл`, `15мл`, `20мл`, `Оригинал` (множественный выбор,
  соответствуют `price_5ml/10ml/15ml/20ml/original`).
- Переключатель режима: **На процент** / **На сумму**.
- Числовое поле значения со знаком (например `+10` или `-5` для процента;
  `+100` или `-50` для суммы в рублях).
- Кнопка «Предпросмотр» → расчёт на клиенте (используя уже загруженные данные
  `selectPerfumes`), таблица: товар | поле | было → стало (для всех затронутых
  товаров × полей; пустые/null поля у товара пропускаются).
- Кнопка «Применить» — отправляет запрос на сервер, после успеха закрывает
  модалку, показывает тост/сообщение «Обновлено N товаров», обновляет список
  (`router.refresh()` + перезагрузка `/api/admin/catalog`).

Расчёт цены:
- Процент: `new = round(old * (1 + value/100))`
- Сумма: `new = old + value`
- Результат клампится к `>= 0` (`Math.max(0, ...)`), округляется до целого ₽
  (`Math.round`).
- Товары, у которых выбранное ценовое поле `null`, пропускаются (не создаём
  цену из ничего).

### API: `POST /api/admin/catalog/bulk-price`

Новый route `src/app/api/admin/catalog/bulk-price/route.ts`.

Тело запроса:
```ts
{
  ids: string[];           // id товаров
  fields: string[];        // подмножество ['price_5ml','price_10ml','price_15ml','price_20ml','price_original']
  mode: 'percent' | 'fixed';
  value: number;           // знаковое
}
```

Логика:
1. Валидация: `ids.length > 0`, `fields` — непустое подмножество допустимых
   полей, `mode` ∈ {percent, fixed}, `value` — конечное число.
2. Для каждого `id` — `getPerfumeById`, для каждого поля из `fields`, если
   текущее значение не `null`, вычислить новое (формула выше), собрать `patch`.
3. Если `patch` непустой — `patchPerfume(id, patch)`.
4. `revalidatePath('/catalog')`.
5. Ответ: `{ ok: true, updated: number }`.

Расчёт цены вынести в чистую функцию `applyPriceDelta(old: number, mode, value): number`
в `src/lib/admin/catalog-logic.ts` (переиспользуется и для клиентского
предпросмотра — импортировать на клиенте, функция чистая, без `server-only`).

## 3. Дашборд — редизайн + алерты + топы

### Визуальный редизайн (`src/app/admin/page.tsx`)
- Карточки статистики: иконки (emoji, в духе остального проекта — без новых
  зависимостей), акцент `gold-500` на ключевых цифрах, более аккуратная сетка
  `grid-cols-2 sm:grid-cols-3 lg:grid-cols-4`.

### Алерты по каталогу
Новая секция «Требует внимания» — карточки-ссылки, каждая с количеством и
переходом в `/admin/catalog?alert=...`:
- **Без фото** (`alert=no-photo`) — `images` пусто/null.
- **Без цен или с нулевыми ценами** (`alert=no-price`) — все из
  `price_5ml/10ml/15ml/20ml/price_original` либо `null`, либо `0`.
- **Нет в наличии** (`alert=out-of-stock`) — `inStock === false`.

Если по категории 0 товаров — карточка не рендерится (или рендерится
приглушённой, без ссылки). Вычисляется на сервере из уже загруженного
`selectPerfumes()`.

### Топ товаров и брендов
Новая секция с двумя колонками: «Топ-5 товаров» и «Топ-5 брендов» по
суммарному количеству позиций в `order_items` за всё время (без ограничения
по дате — проще и достаточно для MVP).

Новая функция в `supabase-server.ts`:
```ts
export async function selectAllOrderItems(): Promise<Record<string, unknown>[]>
```
— `GET /rest/v1/order_items?select=perfume_id,perfume_name,brand,quantity`.

Агрегация — в JS на сервере (дашборд `force-dynamic`, размер таблицы
небольшой): группировка по `perfume_id` (fallback на `perfume_name`, если
`perfume_id` null после `set null` от удалённого товара) и по `brand`,
суммирование `quantity`, сортировка по убыванию, top 5.

Если `order_items` пуст — секция показывает «Пока нет данных».
